◆ T_HAL_USB_REQUEST_BLOCK

typedef struct _hal_usb_request_block T_HAL_USB_REQUEST_BLOCK

USB request block–USB data transfer entity.

Parameters
lengthThe length of data that will be sent or received.
actualThe actual length of data that has been sent or received.
bufThe buffer that stores data will be sent or has been received.
zlpSend a zero-length packet when the length is equal to the maximum packet size of the endpoint.
complete_in_isr1: The complete callback will be called in ISR.
0: The complete callback will not be called in ISR, and the URB will be transferred through HAL_USB_COMMON_ISR_XFER_DONE.
statusThe status of the data transfer.
ep_handleThe handle of endpoint that will transfer data.
completeThe callback will be called when data that has been sent or received.
For sending data, this callback is mainly used to indicate the result of data sending.
For receiving data, this callback is used to get the data already received and related information.
privThe private data.

◆ T_HAL_USB_ISO_PKT_INFO

typedef struct _hal_usb_iso_pkt_info T_HAL_USB_ISO_PKT_INFO

USB isochronous packet information.

For isochronous transfer, multiple packets can be sent once, for example, if usb speed is high speed,
and bInterval of isochronous endpoint is 3, host will send TOKEN packet per 0.5ms, but upper application
may prepare data per 2 ms. In this case, upper application can process 4 packets once.

Parameters
offsetoffset of current packet in buf of T_HAL_USB_ISO_REQUEST_BLOCK
actualactual length of current packet
statusstatus of current packet transferred

◆ T_HAL_USB_ISO_REQUEST_BLOCK

typedef struct _hal_usb_iso_request_block T_HAL_USB_ISO_REQUEST_BLOCK

USB isochronous request block.

For isochronous transfer, ping-pong buffers are used to transfer data, for example, if current data transfer uses buf0,
the next data transfer will use buf1, and vice versa. Data process interval depends on both bInterval in endpoint descriptor and buf_proc_intrvl: 2^(bInterval - 1) * buf_proc_intrvl *125us

Parameters
buf0buffer 0 of ping-pong buffers
buf1buffer 1 of ping-pong buffers
data_per_framedata length per host polling
buf_proc_intrvlnumber of host polling per data process
pkt_cntpacket number per data process
iso_pkt0packets information stored in buf0, refer to T_HAL_USB_ISO_PKT_INFO
iso_pkt1packets information stored in buf1, refer to T_HAL_USB_ISO_PKT_INFO
ep_handlehandle of endpoint that will transfer data
completethis callback will be called when data that has been sent or received.
For sending data, upper application SHOULD prepare data in buf indexed by proc_buf_num.
For receiving data, upper application SHOULD process data stored in buf indexed by proc_buf_num..
privprivate data

int hal_usb_ep_stall_set ( void *  ep_handle,
int  err 
)

halt the interrupt/bulk endpoints or return STALL in the IN status stage of control transfer.
For endpoint0, the api is used when request is not supported or other errors occur,
the device will return STALL in the IN status stage of control transfer.
For non zero endpoint, the api is used when process SetFeature(ENDPOINT_HALT) request

Parameters
ep_handleendpoint handle, , returned by hal_usb_ep_handle_get
errerror that causing stall
Returns
int result, refer to "errno.h"
Example
//endpoint0
int setup_cb(uint8_t *data)
{
if request is not supported
{
hal_usb_ep_stall_set(ep0_handle, -ENOTSUPP);
}
return -ENOTSUPP;
}
//interrupt/bulk endpoints
* int setup_cb(uint8_t *data)
{
if request is SetFeature(ENDPOINT_HALT)
{
uint8_t addr = UGETW(wIndex);
void *ep_handle = hal_usb_ep_handle_get(addr);
hal_usb_ep_stall_set(ep_handle, 0);
}
return 0;
}
