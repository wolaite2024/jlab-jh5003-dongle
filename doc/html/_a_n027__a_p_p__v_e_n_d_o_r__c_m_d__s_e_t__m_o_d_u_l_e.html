Glossary

1 Introduction

APP Vendor Command Set Module is used for transmission between two devices by binary commands. The two devices can be connected via Bluetooth SPP/iAP/LE, or use a UART serial port to transmit data.

The remote device can send commands to MCU(RTL87X3D/RTL87X3E/RTL87X3G) to execute the corresponding action. MCU will also send events to indicate remote device about command status, Bluetooth connection status, DSP status, etc.

1.1 Protocol Format

The message field also includes Sync word, Sequence Number, Length, Command/Event opcode, Parameters, and Checksum.

  • Sync word: size 1 byte, synchronization word is fixed to 0xAA.
  • Sequence Number: size 1 byte, the sequence number should plus 1 after sending every message packet, value is during 1~255.
  • Length: size 2 byte, message payload data length, value is during 1~65535.
  • Command/Event opcode: size 2 byte, command or event opcode, please see commands and events for the details.
  • Parameters: size N byte, option parameter based on command/event opcode.
  • Checksum: size 1 byte, (0xFF - Summation from Sequence Number to parameters + 1).

The Checksum field is only valid for the UART path, no need when transmitting data via Bluetooth SPP, iAP, or LE. The protocol will use Little-Endian and its format that every type of field will be shown as below:

1.2 Processing Flow of APP Vendor Command

  1. Remote devices can send vendor commands to MCU via Bluetooth SPP, iAP, LE, or UART.
  2. When MCU received vendor command, it will handle the command by command index and command path in app_cmd_handler().
  3. After the MCU has finished processing the command, it will report EVENT_ACK, to let the remote device know that the command has been handled, and additional events, to transmit data if necessary.
  4. For the MCU to report the next event, the remote device must provide CMD_ACK to let MCU know that the event (except EVENT_ACK) has been successfully received.

3.2 APP Command Handler

  • Received commands will be parsed and sent to app_cmd_handler().
  • When a command is received, EVENT_ACK will be replied to the remote device.
  • If a duplicate sequence number is checked, only EVENT_ACK will be replied to the remote device. Otherwise, users can use app_report_event() to report other events.
  • Commands are divided into several categories and handled by each module:
    • General
    • Bluetooth
    • HFP
    • AVRCP
    • PBAP
    • Data Transfer
    • TTS
    • Audio
    • EQ
    • Data Capture
    • Sensor
    • Device
    • OTA
    • Key Remap
    • Listening Mode
    • ANC
    • APT
    • One-Wire UART
    • Customized
    • Hearing Aid (HA)
    • Others
  • For example, CMD_MMI is in the category of General commands:
    void app_cmd_handler(uint8_t *cmd_ptr, uint16_t cmd_len, uint8_t cmd_path, uint8_t rx_seqn, uint8_t app_idx)
    {
    uint16_t cmd_id;
    uint8_t ack_pkt[3];
    cmd_id = (uint16_t)(cmd_ptr[0] | (cmd_ptr[1] << 8));
    ack_pkt[0] = cmd_ptr[0];
    ack_pkt[1] = cmd_ptr[1];
    ack_pkt[2] = CMD_SET_STATUS_COMPLETE;
    /* check duplicated seq num */
    if ((cmd_id != CMD_ACK) && (rx_seqn != 0))
    {
    uint8_t *check_rx_seqn = NULL;
    uint8_t report_app_idx = app_idx;
    if (cmd_path == CMD_PATH_UART)
    {
    check_rx_seqn = &uart_rx_seqn;
    report_app_idx = 0;
    }
    else if (cmd_path == CMD_PATH_LE)
    {
    check_rx_seqn = &app_db.le_link[app_idx].rx_cmd_seqn;
    }
    else if ((cmd_path == CMD_PATH_SPP) || (cmd_path == CMD_PATH_IAP))
    {
    check_rx_seqn = &app_db.br_link[app_idx].rx_cmd_seqn;
    }
    if (check_rx_seqn)
    {
    if (*check_rx_seqn == rx_seqn)
    {
    app_report_event(cmd_path, EVENT_ACK, report_app_idx, ack_pkt, 3);
    return;
    }
    *check_rx_seqn = rx_seqn;
    }
    }
    ...
    switch (cmd_id)
    {
    case CMD_ACK:
    {
    uint16_t event_id = (uint16_t)(cmd_ptr[2] | (cmd_ptr[3] << 8));
    if (cmd_path == CMD_PATH_UART)
    {
    app_transfer_queue_recv_ack_check(event_id, cmd_path);
    }
    else if ((cmd_path == CMD_PATH_LE) || (cmd_path == CMD_PATH_SPP) || (cmd_path == CMD_PATH_IAP))
    {
    uint8_t status = cmd_ptr[4];
    app_transfer_queue_recv_ack_check(event_id, cmd_path);
    ...
    }
    }
    break;
    ...
    case CMD_MMI:
    {
    app_cmd_general_cmd_handle(cmd_ptr, cmd_len, cmd_path, app_idx, ack_pkt);
    }
    break;
    ...
    }
    }

4.2 APP Report Event

When users want to send some data to a remote device, there are three ways to achieve this:

4.2.1 APP Report Event via a Specific Path

app_report_event() is used to report events via a specific path.

  • Report a UART event: app_report_uart_event().
    • app_util_calc_checksum() is used to calculate payload checksum for UART events.
  • Report an LE event: app_report_le_event().
  • Report an SPP event: app_report_spp_event().
  • Report an iAP event: app_report_iap_event().

All of the above functions call app_transfer_push_data_queue() to send out the data.

void app_report_event(uint8_t cmd_path, uint16_t event_id, uint8_t app_index, uint8_t *data, uint16_t len)
{
switch (cmd_path)
{
case CMD_PATH_UART:
app_report_uart_event(event_id, data, len);
break;
case CMD_PATH_LE:
if (app_index < MAX_BLE_LINK_NUM)
{
app_report_le_event(&app_db.le_link[app_index], event_id, data, len);
}
break;
case CMD_PATH_SPP:
case CMD_PATH_IAP:
if (app_index < MAX_BR_LINK_NUM)
{
app_report_spp_iap_event(&app_db.br_link[app_index], event_id, data, len, cmd_path);
}
break;
default:
break;
}
}

5 APP Transfer Module Overview

This module is used to push and pop data from the transfer queue. The reference file is shown below:

  • Project source code directory: sdk\src\sample\rws\app_transfer.c

5.1 APP Transfer Data Process

  1. When an event is expected to send out via any command path, app_transfer_push_data_queue() is used to put data into the transfer queue.
  2. When the specific command path is available, app_transfer_pop_data_queue() is used to send data.
  3. After receiving CMD_ACK for an event that has been sent out to a remote device, app_transfer_queue_recv_ack_check() is triggered to switch to the next data in the transfer queue.
bool app_transfer_push_data_queue(uint8_t cmd_path, uint8_t *data, uint16_t data_len, uint8_t idx)
{
T_DT_QUEUE dt_queue = {0};
app_transfer_get_path_handle(cmd_path, &dt_queue);
if (dt_queue.w_dat->cmd_path == CMD_PATH_NONE)
{
dt_queue.w_dat->cmd_path = cmd_path;
dt_queue.w_dat->pkt_ptr = data;
dt_queue.w_dat->pkt_len = data_len;
dt_queue.w_dat->idx = idx;
dt_queue.ctrl->w_idx++;
if (dt_queue.ctrl->w_idx >= dt_queue.fifo_counts)
{
dt_queue.ctrl->w_idx = 0;
}
app_transfer_pop_data_queue(cmd_path);
return true;
}
return false;
}

This module is used to calculate the transfer payload checksum. The reference file is shown below:

    uint8_t app_util_calc_checksum(uint8_t *dataPtr, uint16_t len)
    {
    uint8_t check_sum = 0;
    while (len)
    {
    check_sum += *dataPtr;
    dataPtr++;
    len--;
    }
    return (uint8_t)(0xff - check_sum + 1); //((~check_sum)+1);
    }
